# 매매 중단/재개 정책 (Trading Halt/Resume Policy)

**문서 번호**: OPS-001
**버전**: 1.0
**최종 수정**: 2026-04-04
**승인자**: 운영책임자

## 1. 목적

시스템 이상, 시장 급변, 리스크 한도 초과 시 자동/수동 매매 중단 및 재개 절차를 정의합니다.

## 2. 중단 트리거

### 2.1 자동 중단 (시스템)

| 트리거 | 조건 | 중단 범위 | 복구 조건 |
|--------|------|----------|----------|
| 일일 손실 한도 | 포트폴리오 -3% 이상 | 전체 매매 중단 | 익일 장 개시 시 자동 해제 |
| 주간 손실 한도 | 포트폴리오 -5% 이상 | 전체 매매 중단 | 운영책임자 수동 승인 |
| 주문 연속 실패 | 3회 연속 실패 | 해당 종목 중단 | 원인 분석 후 수동 해제 |
| API 연결 장애 | KIS API 응답 없음 30초 | 전체 매매 중단 | API 정상 복구 확인 |
| 게이트 블록 | 리스크 게이트 BLOCK | 해당 파이프라인 중단 | 게이트 조건 충족 |
| 환율 급변 | USD/KRW ±2% 이내 5분 | USD 자산 매매 중단 | 환율 안정화 확인 |

### 2.2 수동 중단

- **운영자 긴급 중단**: Telegram `/halt` 명령 또는 API `POST /api/system/halt`
- **정기 점검**: 스케줄러에 의한 사전 예약 중단
- **외부 요인**: 거래소 서킷브레이커, 시장 전체 중단 시

## 3. 중단 절차

```
1. 트리거 감지
   ↓
2. PipelineStateMachine → HALTED 전이
   ↓
3. 미체결 주문 일괄 취소 (KIS API cancel_order)
   ↓
4. 감사 로그 기록 (AuditLogger: TRADING_HALTED, 사유)
   ↓
5. 알림 발송 (Telegram + 이메일)
   ↓
6. 운영 대시보드 상태 갱신
```

## 4. 재개 절차

```
1. 중단 사유 해소 확인
   ↓
2. 시스템 상태 점검 (DB, Redis, KIS API 연결 확인)
   ↓
3. 미체결 주문 상태 동기화
   ↓
4. 운영책임자 승인 (주간 손실 한도 초과 시 필수)
   ↓
5. PipelineStateMachine → IDLE 전이
   ↓
6. 감사 로그 기록 (TRADING_RESUMED, 승인자)
   ↓
7. 알림 발송 (매매 재개 통지)
```

## 5. 에스컬레이션

| 단계 | 시간 | 대상 | 액션 |
|------|------|------|------|
| L1 | 즉시 | 시스템 (자동) | 매매 중단 + 알림 |
| L2 | 5분 | 운영 담당자 | 원인 분석 + 판단 |
| L3 | 30분 | 운영책임자 | 재개/연장 결정 |
| L4 | 2시간 | 경영진 | 장기 중단 판단 |

## 6. 코드 연동 포인트

- `core/state_machine.py`: `PipelineState.HALTED` 전이 로직
- `core/pipeline.py`: 게이트 BLOCK 시 HALTED 전이
- `core/order_executor/executor.py`: 미체결 주문 취소 메서드
- `api/routes/system.py`: `/api/system/halt`, `/api/system/resume` 엔드포인트
- `config/settings.py`: 손실 한도 임계값 설정
- `config/operational_thresholds.yaml`: 중단 임계값 운영 설정 (환율 변동폭, 손실 한도 등)
